# 测试说明

1. 准备测试数据  
在数据库里把 1 个菜品库存改大一点（例如 500），并确认它可售、所在食堂在营业时间内。
准备 50~200 个可登录用户 token（至少 50 个）。

2. 新建 `users.csv`  

3. 打开 JMeter，创建测试计划  
Test Plan
 └─ Thread Group
     ├─ CSV Data Set Config
     ├─ HTTP Header Manager
     ├─ HTTP Request
     ├─ Synchronizing Timer
     ├─ JSON Extractor（可选）
     ├─ Response Assertion（可选）
     ├─ View Results Tree（调试时开）
     ├─ Summary Report
     └─ Aggregate Report

4. 配置 CSV Data Set  
  Filename：D:\apache-jmeter-5.6.3\bin\users.csv
  File Encoding：UTF-8
  Variable Names：token,longitude,latitude,dishId,qty
  Ignore first line：True
  Delimiter：,
  Recycle on EOF：False
  Stop thread on EOF：True
  Sharing mode：All threads

5. 配置 HTTP Header Manager
`authorization: ${token}`  
`Content-Type: application/json`
`Accept: application/json`

6. 配置下单请求 Request 
`POST http://127.0.0.1:8082/food-order`  
Body（Raw + JSON）：
```json
{
  "items":[
    {"dishId":101,"quantity":1},
    {"dishId":102,"quantity":1}
  ],
  "remark":"deadlock-A-${__threadNum}",
  "longitude":${longitude},
  "latitude":${latitude}
}
```

7. 过程
- **Threads（线程数）**：`5`->`25`->`50`->`100`->`200`
- **Ramp-Up**：`1`
- **Loop Count**：`1`
- **Synchronizing Timer - Group by**：`5`->`25`->`50`->`100`->`200`
- **Synchronizing Timer - Timeout**：`3000`

8. 核心指标（在 Aggregate Report）  
`Error %`：整体失败率
`Throughput`：吞吐（每秒请求数）
`95% Line`：P95 延迟（用户体验关键）
`Max`：最慢请求

9. 做正确性复核
执行 SQL：
```sql
select id,quantity from tb_dish where quantity < 0;
```
结果必须 0 行（不能负库存）。  
再看下单总数与库存减少是否大致一致，确认没有并发错单。

10. 快速判断结论  
可用：`Error% < 1%`、`P95 < 500ms`、无负库存。

- 预期：不会出现线程永久卡住；最多是拿锁失败返回“too many requests…”。

  5+5、25+25、50+50（库存一致 顺利下单）、100+100（175/200部分返回失败但没有死锁）

**1) 先明确本项目的并发敏感点（从代码出发）**
- 下单扣库存：`/food-order` → FoodOrderController.java 调到 FoodOrderServiceImpl.createOrder  
  - 同一用户防重复提交锁：见 FoodOrderServiceImpl.java（`lock:food:order:create:user:{uid}`）  
  - 按菜品加库存锁 + 排序防死锁：同方法内（`lock:dish:stock:{dishId}`）  
  - 库存扣减是单条 SQL 原子条件更新：见 FoodOrderServiceImpl.java
  - 下单额外约束：必须带经纬度 + 距离校验 + 营业时间：见 FoodOrderServiceImpl.java
- 取消订单 / 管理员改状态：同一把“订单锁”串行化：  
  - 用户取消：FoodOrderServiceImpl.cancelMyOrder（`lock:food:order:{orderId}`）  
  - 管理员改状态：FoodOrderServiceImpl.adminUpdateStatus
- 热点订单详情缓存的“互斥重建”（防击穿）：见 FoodOrderServiceImpl.java 调用 CacheClient.queryWithMutex（Redis `SETNX` 锁见 CacheClient.java）
- 评价并发：同一用户对同一菜品“插入冲突转更新”（依赖唯一键约束）：见 DishReviewServiceImpl.submitReview

# 测试结果

## 1）# Samples

表示**总请求数**。

比如你线程数 100，Loop=1，那理论上：

```
# Samples = 100
```

也就是一共发了 100 次请求。

------

## 2）Average

表示**平均响应时间**，单位毫秒。

比如：

```
Average = 120 ms
```

意思是平均每个请求 120ms 返回。

### 怎么理解

- 越小越好
- 但你的场景里，正确性比极致低延迟更重要

------

## 3）Min

最短响应时间。

表示最快的那个请求有多快。

------

## 4）Max

最长响应时间。

表示最慢的那个请求有多慢。

### 你这里特别要看

如果 Max 特别大，说明高并发下有些请求在等锁、等数据库、等资源。

------

## 5）Std. Dev.

标准差，表示响应时间波动程度。

### 怎么理解

- 越小：说明请求响应比较稳定
- 越大：说明有些很快，有些很慢，系统波动大

你的场景里波动大很正常，因为有人很快抢到，有人卡在锁竞争上。

------

## 6）Error %

错误率。

### 这是最容易误解的指标

在你的场景里：

**不是所有失败都叫系统错误。**

你要区分：

### 合理失败

比如：

- 库存不足
- 抢锁失败
- 并发过高导致业务返回“too many requests”

这种是业务预期内失败，不一定是 bug。

### 非合理失败

比如：

- HTTP 500
- 空指针
- JSON 解析错误
- 数据库异常
- 超时无响应

这种才是问题。

所以：

> 你不能只看 Error%，还要看失败内容是什么。

------

## 7）Throughput

吞吐量，表示单位时间内处理了多少请求。

通常单位是：

```
requests/second
```

### 怎么理解

比如 Throughput = 80/sec，表示系统平均每秒处理 80 个请求。

但你这个场景库存只有 50，抢完就结束，所以吞吐量只是参考，不是唯一目标。

------

## 8）Received KB/sec / Sent KB/sec

网络收发速率。

这个一般不是你当前最核心关注点。
